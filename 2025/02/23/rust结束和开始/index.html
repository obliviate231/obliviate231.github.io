修改 ThreadPool 存放 Worker 实例而不是直接存放线程

第11行, 我们留下了个关于创建线程的注释。如何实际创建线程呢？这是一个难题。

thread::spawn，它期望获取一些一旦创建线程就应该执行的代码。然而，我们希望开始线程并使其等待稍后传递的代码。标准库的线程实现并没有包含这么做的方法；我们必须自己实现。

修改 ThreadPool 来储存一个传输 Job 实例的发送者

  1. ThreadPool 会创建一个信道并充当发送者。
  2. 每个 Worker 将会充当接收者。
  3. 新建一个 Job 结构体来存放用于向信道中发送的闭包。
  4. execute 方法会在发送者发出期望执行的任务。
  5. 在线程中，Worker 会遍历接收者并执行任何接收到的任务

注意多个生产者, 单消费者用Arc<Mutex> 智能指针解决问题. 使用 Arc 和 Mutex 在 worker 间共享接收者

为存放每一个闭包的 Box 创建一个 Job 类型别名，接着在信道中发出任务

job “类型别名用来创建类型同义词”

  • type 关键字：在 Rust 里，type 关键字用于创建类型别名，也就是给一个已有的复杂类型起一个新的、更简洁的名字，这样可以提高代码的可读性和可维护性。

  • Job：这是我们定义的类型别名，后续代码中就可以直接使用 Job 来代表这个复杂的类型。

  • Box<...>：Box 是 Rust 中的智能指针，它会在堆上分配内存来存储数据，然后在栈上存储一个指向该数据的指针。使用 Box 主要是为了处理动态大小的类型，因为 Rust 要求栈上的变量大小在编译时必须是确定的。

  • dyn FnOnce()

    ：

    • dyn 关键字用于表示这是一个动态分发的 trait 对象。
    • FnOnce 是 Rust 中的一个 trait，它表示一个可以被调用一次的闭包或函数。FnOnce() 意味着这个闭包或函数不接受任何参数，并且只可以被调用一次。

  • Send：Send 也是一个 trait，它表示实现了该 trait 的类型可以安全地在线程间传递。当我们要把一个任务从一个线程发送到另一个线程去执行时，这个任务的类型就必须实现 Send trait。

  • 'static：'static 是一个生命周期注解，它表示这个类型的生命周期和整个程序的运行周期一样长。在这个上下文中，意味着闭包所捕获的所有引用的生命周期都必须是 'static，这样才能保证闭包可以安全地在线程间传递和存储。

这里的错误告诉我们并不能调用 join，因为我们只有每一个 worker 的可变借用，而 join 需要获取其参数的所有权。为了解决这个问题，需要一个方法将 thread 移动出拥有其所有权的 Worker 实例以便 join 可以消费这个线程。示例 17-15 中我们曾见过这么做的方法：如果 Worker 存放的是 Option<thread::JoinHandle<()>，就可以在 Option 上调用 take 方法将值从 Some 成员中移动出来而对 None 成员不做处理。换句话说，正在运行的 Worker 的 thread 将是 Some 成员值，而当需要清理 worker 时，将 Some 替换为 None，这样 worker 就没有可以运行的线程
